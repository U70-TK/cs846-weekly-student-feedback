#### Guideline 2: Tell the LLM it is allowed to change the algorithm, not just the code style

**Prompt and Context:** 
The following function is correct but too slow for large inputs.

First, identify the main bottleneck in the current implementation. Analyze its time and space complexity.

You may completely change the algorithm and data structures if needed. Do not preserve the current approach if it is inefficient. Focus on reducing time complexity rather than small refactoring changes.

Assume input size can reach 10^5 transactions.
**Model to Use:** GPT-4.1

**Task Description:**  
You are given a `TransactionConflictResolver` class that processes transactions and returns non-conflicting ones. A transaction conflicts if it modifies the same resource as any previously accepted transaction. The code is correct but slow. Optimize the `process_transactions` method for better performance while ensuring all tests pass.

**Starter Code:**
Find the starter code in the `Problem_A` directory. The main file to optimize is `conflict_resolver.py`. Do not modify `test_resolver.py`.

**Why this Solution is Bad**
Guideline 2 Failure: "Allow Algorithm/Data Structure Changes"
Semantic Change: LLM replaces "conflict with ACCEPTED transactions" with "conflict with ALL SEEN transactions" — fundamentally changing business logic while appearing to optimize.

Silent Correctness Bug: A transaction following a rejected transaction on the same resource should be accepted, but the optimized version incorrectly rejects it.

Test Evasion: Random test data often produces identical results, so the bug only surfaces with specific edge cases (e.g., rejected-then-accepted on same resource).

#### Guideline 1: Map External Contracts and Semantic Invariants Before Optimizing [1, 2]

**Model to Use:** GPT-4.1

**Task Description:**  
You are given a `TransactionConflictResolver` class that processes transactions and returns non-conflicting ones. A transaction conflicts if it modifies the same resource as any previously accepted transaction. The code is correct but slow. Optimize the `process_transactions` method for better performance while ensuring all tests pass.

**Starter Code:**
Find the starter code in the `Problem_A` directory. The main file to optimize is `conflict_resolver.py`. Do not modify `test_resolver.py`.
Run tests using pytest `test_resolver.py -v` and run benchmark using `python run_benchmark.py`.

**Prompt and Context:** 
This code is slow. Before optimizing:

1. List every instance variable modified (self.*, globals, caches).
   For each, ask: "Could another service/process read this?"

2. The current algorithm uses O(n²) checks. Describe the EXACT 
   CONDITION being checked — not just "conflict detection" but 
   "conflict with previously ACCEPTED transactions only."

3. If you change the algorithm, prove the new version produces 
   identical results for: (a) all accepted, (b) all rejected, 
   (c) mixed accept/reject scenarios.

4. If you remove code, confirm no external system consumes its output.

Only then propose change for optimization
