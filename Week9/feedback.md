#### Guideline 4: Decompose Complex Methods Before Asking for Tests

**Prompt and Context:** 

```
You are helping me write unit tests. Follow Guideline 4: Decompose Complex Methods Before Asking for Tests.

Target: The function clamp(value, lo, hi) in clamp.py (code below).

---
def clamp(value: float, lo: float, hi: float) -> float:
    """Return value clamped to [lo, hi]. Assumes lo <= hi (no validation)."""
    if value < lo:
        return lo
    if value > hi:
        return hi
    return value
---

Step 1: List the distinct behaviors in this function (e.g. validation, calculation, edge handling). Give a short list.

Step 2: For each behavior, generate separate focused pytest tests (one responsibility per test group). Use @pytest.mark.parametrize where appropriate.

Step 3: For each behavior, include boundary + negative cases (nulls, invalid inputs, exception paths) where applicable.

Constraints:
- Do not modify clamp.py.
- Use clear assertions.
- Use descriptive test names.

Return the full pytest test file only.
```

**Task:**

Write pytest unit tests for the function `clamp(value, lo, hi)` in `clamp.py`. Do not modify clamp.py. Use pytest with @pytest.mark.parametrize where appropriate, cover boundary cases (below lo, above hi, equal to lo, equal to hi, and strictly between lo and hi), and use strong assertions that check exact expected return values; deliver a single runnable test file with clear test names.

**Solution or Output generated by Github Copilot:**

from clamp import clamp

@pytest.mark.parametrize("value, lo, hi, expected", [
    # value below lo
    (-1, 0, 10, 0),
    (-100, -50, -10, -50),
    (4.9, 5, 10, 5),
    # value above hi
    (11, 0, 10, 10),
    (0, -50, -10, -10),
    (10.1, 5, 10, 10),
    # value in range
    (5, 0, 10, 5),
    (-20, -50, -10, -20),
    (7.5, 5, 10, 7.5),
    # value == lo
    (0, 0, 10, 0),
    (-50, -50, -10, -50),
    (5, 5, 10, 5),
    # value == hi
    (10, 0, 10, 10),
    (-10, -50, -10, -10),
    (10, 5, 10, 10),
])
def test_clamp_contract(value, lo, hi, expected):
    assert clamp(value, lo, hi) == expected

# Fault Model FM1: Lower bound uses <= instead of <
def test_clamp_lower_bound_strict():
    # If bug exists, value == lo would return lo+1 or not lo
    assert clamp(0, 0, 10) == 0
    assert clamp(-50, -50, -10) == -50
    assert clamp(5, 5, 10) == 5

# Fault Model FM2: Upper bound uses < instead of >
def test_clamp_upper_bound_strict():
    # If bug exists, value == hi would return hi-1 or not hi
    assert clamp(10, 0, 10) == 10
    assert clamp(-10, -50, -10) == -10
    assert clamp(10, 5, 10) == 10

# Fault Model FM3: Returns value when value < lo (forgets to clamp low)
@pytest.mark.parametrize("value, lo, hi", [
    (-1, 0, 10),
    (-100, -50, -10),
    (4.9, 5, 10),
])
def test_clamp_clamps_low(value, lo, hi):
    assert clamp(value, lo, hi) == lo

```

**Why this is bad:**

1. **Decomposition is forced on single-responsibility code.**
`clamp()` has one responsibility: return value clamped to [lo, hi]. There are no separate “validate → transform → persist” steps. Guideline 4 asks the model to list distinct behaviors and test each one, so it invents behaviors like “below bound,” “above bound,” “in range,” and “validation.” Those are input cases for the same behavior, not separate responsibilities, which leads to redundant test groups and noisy structure.

2. **The model is steered toward invented structure instead of real coverage.**
The guideline pushes the model to create a test group per “behavior.” The result is many small test methods that could be a few parametrized tests. The model prioritizes structure (multiple classes and methods) over clarity and over targeting real faults (e.g. off-by-one at boundaries). So the test suite looks organized but is shallow and may not catch simple bugs like <= vs <.

3. **“Negative cases per behavior” invents a contract the code doesn’t have.**
The spec says the function assumes lo <= hi and does not validate. Following “boundary + negative per behavior,” the model adds tests for None and for lo > hi. That invents a validation responsibility the code doesn’t have, so tests are either underspecified (e.g. for lo > hi) or irrelevant (e.g. for None), and they don’t reflect the documented contract.

The model may never produce a compact, fault-focused suite, may add tests for non-existent validation, and may miss boundary bugs. So Guideline 4, as applied here, does not work as intended for this question.

## 3. New and Updated Guidelines that worked

### Guideline 1: Stress-Test with Concurrency Invariant Assertions [1][2]
**Task Description**

Open `bank.py`. use (GPT 4.1)  to generate **pytest tests** that use multiple threads for every test, with no single-threaded tests allowed. Each test should define a system-wide invariant (such as conservation, accuracy, liveness, or consistency), spawn 10-20 threads performing concurrent operations, and assert that the invariant holds across 50 repeated iterations. Run the generated tests and record how many of the five concurrency bugs are detected.

**Prompt and Context**
```
Generate pytest tests for `bank.py`. All tests MUST use multiple
threads, no single-threaded tests. Use `threading.Barrier` to start threads simultaneously. Loop each scenario 50 times, fail if any iteration breaks.

Tests:
1. 10 threads do transfer(a,b,1) + transfer(b,a,1) x100.
   Assert a.balance + b.balance == original total.

2. 20 threads each do 1 transfer.
   Assert get_total_transfers() == 20.

3. 5 threads do A→B, 5 do B→A in a loop.
   join(timeout=5). Fail if any thread still alive.

4. 10 threads deposit(10) x50 into one account.
   Assert balance == 5000.

5. After N concurrent transfers:
   Assert len(get_log()) == get_total_transfers().
```

**Solution or Output generated by GitHub Copilot:**
```
### Multi-threaded Test Results

The following tests were executed to validate the thread-safe behavior of `bank.py`:

1. **Concurrent Transfers:**
   - **Description:** 10 threads performed `transfer(a, b, 1)` and `transfer(b, a, 1)` 100 times each.
   - **Result:** Passed. The total balance across accounts remained constant.

2. **Total Transfers Validation:**
   - **Description:** 20 threads each performed one transfer.
   - **Result:** Passed. The total number of transfers matched the expected count.

3. **Bidirectional Transfers:**
   - **Description:** 5 threads transferred from A→B, and 5 threads transferred from B→A in a loop.
   - **Result:** Passed. No threads were left alive after the timeout.

4. **Concurrent Deposits:**
   - **Description:** 10 threads deposited $10 fifty times into one account.
   - **Result:** Passed. The final balance matched the expected value of $5000.

5. **Transfer Log Validation:**
   - **Description:** After N concurrent transfers, the length of the log was compared to the total number of transfers.
   - **Result:** Passed. The log length matched the total transfers.

```

**Why This Guideline Is Good Here:**

- **Targets thread interleaving, not input space:** `threading.Barrier` forces all threads to enter the critical section simultaneously, maximizing contention on shared locks and exposing race windows that sequential tests cannot reach.

- **Zero false positives:** All five tests assert properties derived from the code’s actual synchronization contract (lock-protected mutations and atomic transfers), unlike Guideline 3 which asserted nonexistent input validation and produced four spurious failures.

- **Invariant-based oracles handle non-determinism:** Instead of asserting exact return values (which depend on scheduler ordering), the tests verify relational properties (e.g., `a.balance + b.balance == total`) that must hold under any valid thread interleaving.

- **Maps directly to concurrency bug taxonomy:** Each test targets a distinct bug class from the standard classification [3]: data race (INV4), atomicity violation (INV1), deadlock (INV3), lost update (INV2), and unsafe shared state (INV5).

- **Low overhead, no special tooling:** Uses only `threading.Barrier`, `threading.Lock`, and `thread.join(timeout=...)` from the Python standard library. No model checkers or instrumentation are required-just structured stress testing with loop amplification.

#### Guideline 2: Match Test Structure to Code Complexity

**Description:**  

**Do not force decomposition on every method.** First decide whether the method under test has one clear responsibility or several distinct sub-behaviors.

- **Single-responsibility or trivial code** (e.g. pure functions, thin wrappers, one clear “do X”):  
  Ask for **one focused test suite** that covers:
  - Boundary and negative cases (as in Guideline 3 (in-class)),
  - A small **fault model** (as in Guideline 5 (in-class)) if the code has plausible near-correct bugs,
  - Strong, exact assertions.
  Do **not** ask the LLM to “list distinct behaviors” or “generate tests per behavior” for such code.

- **Multi-responsibility code** (e.g. validate → transform → persist → log):  
  Then apply **Guideline 4**: list documentable sub-behaviors, generate separate test groups per sub-behavior, and keep boundary/negative/fault-model within each group.

**Rule of thumb:** If you cannot write 2–3 distinct, non-overlapping sentence descriptions of what the method does (e.g. “validates input”, “computes price”, “saves to DB”), treat it as single-responsibility and skip decomposition.

**Reasoning:** 

1. **Guideline 4’s rationale** (Wang et al. [4]) is that test quality degrades on **long, multi-responsibility** methods because the model loses track of which behavior it is testing. Decomposition helps there. For **short, single-responsibility** methods there is no such confusion—there is only one behavior. Forcing decomposition has no benefit and causes the LLM to invent fake “behaviors” (e.g. “handles below bound”, “handles above bound”) that are just input categories, leading to redundant structure and sometimes invented contracts (e.g. “validation”).

2. **Single-responsibility code** is better tested by:
   - A small set of **boundary/negative** cases, and/or
   - A **fault model** with one test per plausible bug,
   rather than by artificial “behavior” groups.

3. **Consistency and auditability:** When the test structure matches the real structure of the code (one behavior → one suite; many behaviors → one suite per behavior), it is easier to audit coverage and to know what is being tested. Invented behaviors make the test suite misleading.

**Prompt and Context:**

```
You are helping me write unit tests. The target is a single-responsibility function: it does one thing only. Do NOT decompose it into multiple "behaviors." Generate one focused test suite.

Target: clamp(value, lo, hi) in clamp.py

Contract:
- If value < lo, return lo.
- If value > hi, return hi.
- Otherwise return value.
- No input validation (assumes lo <= hi).

Required:
1. Boundary cases with strong assertions (exact expected values):
   - value below lo (e.g. value=-1, lo=0, hi=10 -> 0)
   - value above hi (e.g. value=11, lo=0, hi=10 -> 10)
   - value in range (unchanged)
   - value == lo and value == hi (boundary)

2. Fault model (at least one test each that would fail if the bug exists):
   - FM1: Lower bound uses <= instead of < (e.g. value=0, lo=0, hi=10 must return 0, not lo)
   - FM2: Upper bound uses < instead of > (e.g. value=10, lo=0, hi=10 must return 10)
   - FM3: Returns value when value < lo (forgets to clamp low)

Use pytest; @pytest.mark.parametrize for multiple inputs. One test file only. Clear assertions. Do not add validation tests (e.g. None, lo>hi) unless we extend the contract.

```

Use pytest with @pytest.mark.parametrize where appropriate, cover boundary cases (below lo, above hi, equal to lo, equal to hi, and strictly between lo and hi), and use strong assertions that check exact expected return values; deliver a single runnable test file with clear test names.

import pytest
from clamp import clamp

# Behavior 1: Value below lower bound
@pytest.mark.parametrize("value, lo, hi, expected", [
    (0, 1, 5, 1),
    (-10, -5, 0, -5),
    (-100, -50, -10, -50),
])
def test_clamp_value_below_lo(value, lo, hi, expected):
    assert clamp(value, lo, hi) == expected

# Behavior 2: Value above upper bound
@pytest.mark.parametrize("value, lo, hi, expected", [
    (10, 1, 5, 5),
    (1, -5, 0, 0),
    (100, -50, -10, -10),
])
def test_clamp_value_above_hi(value, lo, hi, expected):
    assert clamp(value, lo, hi) == expected

# Behavior 3: Value within bounds (including at boundaries)
@pytest.mark.parametrize("value, lo, hi, expected", [
    (3, 1, 5, 3),
    (1, 1, 5, 1),  # at lower bound
    (5, 1, 5, 5),  # at upper bound
    (-5, -5, 0, -5),  # at lower bound
    (0, -5, 0, 0),  # at upper bound
    (-10, -50, -10, -10),  # at upper bound
    (-50, -50, -10, -50),  # at lower bound
])
def test_clamp_value_within_bounds(value, lo, hi, expected):
    assert clamp(value, lo, hi) == expected

# Behavior 4: No validation for lo > hi (assumes lo <= hi)
def test_clamp_no_validation_lo_gt_hi():
    # Function does not validate lo > hi, so result may be unexpected
    assert clamp(5, 10, 1) == 10  # value < lo, returns lo
    assert clamp(-5, 10, 1) == 10  # value < lo, returns lo
    assert clamp(15, 10, 1) == 10  # value > hi, returns lo

# Behavior 5: Handling of float values and precision
@pytest.mark.parametrize("value, lo, hi, expected", [
    (1.5, 1.0, 2.0, 1.5),
    (0.999999, 1.0, 2.0, 1.0),
    (2.000001, 1.0, 2.0, 2.0),
    (1.0, 1.0, 2.0, 1.0),
    (2.0, 1.0, 2.0, 2.0),
])
def test_clamp_float_precision(value, lo, hi, expected):
    assert clamp(value, lo, hi) == expected

# Behavior 6: Negative cases (invalid types)
@pytest.mark.parametrize("value, lo, hi", [
    (None, 1, 5),
    (3, None, 5),
    (3, 1, None),
    ("a", 1, 5),
    (3, "b", 5),
    (3, 1, "c"),
])
def test_clamp_invalid_types(value, lo, hi):
    with pytest.raises(TypeError):
        clamp(value, lo, hi)
```

**Why is this better?**

For `clamp()`:

- **Guideline 4** forces “list behaviors → test per behavior”. The model invents 4–5 “behaviors”, creates multiple test classes, adds invented “validation” tests, and still may miss a simple boundary bug (e.g. `<=` vs `<`) because no fault-model is requested.

- **New guideline** classifies `clamp()` as single-responsibility (one sentence: “return value clamped to [lo, hi]”). So we ask for:
  - One test suite,
  - Boundary cases (below lo, above hi, in range, value == lo, value == hi),
  - Optional fault model (e.g. FM1: wrong comparison at lower bound; FM2: wrong comparison at upper bound; FM3: wrong return order),
  - Strong assertions (exact expected values).

**Result:** fewer, clearer tests; no invented behaviors; optional fault-model for better defect detection; and no irrelevant “validation” tests unless we explicitly add them to the contract.
